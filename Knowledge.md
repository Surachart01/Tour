# Knowledge Base: Tour Booking Management System

## ภาพรวม (System Overview)

ระบบนี้เป็นแพลตฟอร์มจัดการการท่องเที่ยวแบบ B2B ที่เชื่อมระหว่าง **Agent** (ผู้จัดทัวร์ระดับกลาง) กับ **Admin** (ผู้ดำเนินการหลัก) โดยแต่ละฝ่ายมีสิทธิ์และหน้าที่แตกต่างกัน

---

## 1. กระบวนการจองจาก Agent → Admin (Booking Flow)

### 1.1 ขั้นตอนฝั่ง Agent

Agent เข้าสู่หน้า **Agent Portal** และเริ่มสร้าง **Quotation** โดยเลือกบริการต่างๆ ดังนี้:

| บริการ | รายละเอียด |
|--------|-----------|
| 🏨 **Hotel** | เลือกโรงแรมตามเมือง, ประเภทห้อง, วันเข้า-ออก |
| 🚐 **Transfer** | รถรับส่ง: สนามบิน↔โรงแรม, เมือง↔เมือง |
| 🎡 **Excursion** | กิจกรรมท่องเที่ยว เช่น ดำน้ำ, ล่องเรือ, เยี่ยมชมสถานที่ |
| 🗺️ **Tour** | แพ็กเกจทัวร์หลายวัน-หลายเมือง รวมที่พักและรถ |
| ✈️ **Flight** | ตั๋วเครื่องบิน (ถ้ามี) |
| 💰 **Others** | ค่าใช้จ่ายเบ็ดเตล็ด เช่น ค่าวีซ่า, ค่าธรรมเนียม |

เมื่อเลือกครบ → Agent กด **"Convert to Booking"**

### 1.2 สิ่งที่เกิดขึ้นหลัง Convert

1. Quotation ถูกแปลงเป็น **Booking** อย่างเป็นทางการ
2. Admin ได้รับ **Notification** แจ้งเตือนบนทุกหน้า
3. ข้อมูลทั้งหมดจาก Quotation ถูก copy มายัง Booking โดยสมบูรณ์
4. สถานะเปลี่ยนจาก `quotation` → `booking`

## 2. หลักการออกแบบ: ข้อมูลกลับมาเหมือนเดิม (Mirror Structure)

> **หลักการสำคัญ:** ทุกบริการที่ Agent เลือกใน Quotation จะแสดงผล **เหมือนกันทุกประการ** ในหน้า Admin Booking
>
> **ยกเว้น: โรงแรม** — ที่ Admin สามารถเปลี่ยนแปลงได้

เหตุผลที่โรงแรมพิเศษกว่าบริการอื่น:
- โรงแรมที่ Agent เลือกอาจ **เต็มในวันที่ต้องการ** ณ เวลาที่ confirm
- ในสัญญากับลูกค้าระบุว่า: *"หากโรงแรมที่เลือกไม่ว่าง บริษัทจะจัดโรงแรมระดับเดียวกันทดแทน"*
- Admin ต้องมีอำนาจแก้ไขโรงแรมได้ **โดยไม่ยกเลิก booking**
- ไม่ควรยกเลิกทัวร์เพียงเพราะโรงแรมใดโรงแรมหนึ่งเต็ม

## 3. ระบบ Tour — โครงสร้างที่ซับซ้อน (Multi-day Tour Structure)

### 3.1 ลักษณะพิเศษของ Tour

- มี **หลายวัน** (เช่น 7 วัน 6 คืน)
- **หลายจังหวัด** ในทริปเดียว (เช่น เชียงใหม่ → เชียงราย → แม่ฮ่องสอน)
- **แต่ละคืนอยู่คนละโรงแรม** ในคนละเมือง
- **แต่ละเมืองมีตัวเลือกโรงแรมหลายแห่ง** ให้ Agent เลือก

### 3.2 โครงสร้างข้อมูลในระบบ (Data Model)

```
Tour (แพ็กเกจทัวร์ Master)
  └── Tour Days (วันที่ 1, 2, 3, ... N)
        └── Tour Details (โรงแรม, เมือง, ห้องพัก per day)
              ↕ (ถ้า Admin เปลี่ยน)
Tour Trip Item Hotels (ข้อมูล Override เฉพาะ booking นี้)
```

### 3.3 รูปแบบที่แสดงใน Admin Booking (Tour Accommodation Section)

```
┌─────────────────────────────────────────────────────┐
│  ↓ TRANSFER IN  (รถรับ วันแรก - ดึงอัตโนมัติ)         │
│  ─────────────────────────────────────────────────  │
│  🌙 คืนที่ 1: โรงแรม A — เชียงใหม่               │
│  🌙 คืนที่ 2: โรงแรม B — เชียงราย                │
│  🌙 คืนที่ 3: โรงแรม B — เชียงราย                │
│  🌙 คืนที่ 4: โรงแรม C — แม่ฮ่องสอน              │
│  ─────────────────────────────────────────────────  │
│  ↑ TRANSFER OUT (รถส่ง วันสุดท้าย - ดึงอัตโนมัติ)      │
└─────────────────────────────────────────────────────┘
```

## 4. รูปแบบเส้นทางทัวร์ (Tour Routing Patterns)

### Pattern 1: รถขึ้นเหนือ — จบที่เหนือ + บินต่อ
```
กรุงเทพฯ (รถ) → เชียงใหม่ → เชียงราย → [สนามบินเชียงราย] → ✈️ ต่อเครื่อง
```

### Pattern 2: บินไปเหนือ — ลงมา กทม. — ต่อเครื่อง/รถ
```
✈️ BKK → เชียงใหม่ → [ทัวร์] → เชียงราย → (รถ/เครื่อง) → กรุงเทพฯ → ✈️/🚌 ต่อ
```

### Pattern 3: บินไปเหนือ — อยู่เหนือ — บินออกจากเหนือ
```
✈️ BKK → เชียงใหม่ → [ทัวร์วนเหนือ] → [สนามบินเชียงใหม่/เชียงราย] → ✈️ กลับ
```

### Pattern 4: Loop ทัวร์ — ออก-วน-กลับจุดเดิม
```
เชียงใหม่ → แม่ฮ่องสอน → ปาย → เชียงใหม่
```

### Pattern 5: International Connect — ต่อต่างประเทศ
```
BKK → เชียงใหม่ → [ทัวร์] → เชียงราย → ✈️ → เวียดนาม/ลาว/เมียนมา
```

## 5. Transfer In / Transfer Out — ตรรกะอัตโนมัติ

| | ความหมาย | วิธีดึงข้อมูล |
|---|---|---|
| **Transfer In** | รถรับส่งรายการ **แรกสุด** ของทริป | `transfer_trip_items` ORDER BY `from_date` ASC → อันดับ 1 |
| **Transfer Out** | รถรับส่งรายการ **สุดท้าย** ของทริป | `transfer_trip_items` ORDER BY `from_date` ASC → อันดับสุดท้าย |

**Format:** `วันที่ | Pickup: เวลา | จาก → ถึง | (คำอธิบาย)`

---

## 6. ตรรกะการ Override โรงแรม (Hotel Override Logic)

```
ถ้า tour_trip_item_hotels มีข้อมูล (has_overrides = true)
  → ใช้ข้อมูล Override ที่ Admin เลือก ✅
ถ้าไม่มี Override
  → ใช้ข้อมูลจาก Tour Master Definition (ที่ Agent เลือกตอนแรก)
```

- **Non-destructive:** Admin เปลี่ยนโรงแรมโดยไม่ลบข้อมูลเดิม
- **Deduplication:** ระบบป้องกันซ้ำด้วย `Map` keyed by `day` (แต่ละคืนมีโรงแรมได้แค่ 1 แห่ง)

## 7. โครงสร้างความสัมพันธ์ข้อมูล (Data Relationship)

```
trips (Booking Master)
  ├── transfer_trip_items    → Transfer In/Out (ดึงอัตโนมัติ)
  ├── hotel_trip_items       → โรงแรมเดี่ยว (ที่ Agent จอง)
  ├── excursion_trip_items   → กิจกรรม
  ├── flight_trip_items      → เที่ยวบิน
  ├── other_trip_items       → ค่าใช้จ่ายอื่น
  └── tour_trip_items        → แพ็กเกจทัวร์
        └── tour_trip_item_hotels  → โรงแรมแต่ละคืน (Override โดย Admin)
```

---

## 8. หลักการออกแบบสำคัญ (Core Design Principles)

1. **Non-destructive Override** — Admin เปลี่ยนโรงแรมโดยไม่ลบข้อมูลเดิมของ Agent
2. **Transparency** — ทุกการเปลี่ยนแปลงมี Reason for Change เสมอ
3. **Auto-derive** — ข้อมูลที่หาได้อัตโนมัติ (เช่น Transfer In/Out) ไม่ต้องกรอกมือ
4. **No-cancel on Full Hotel** — โรงแรมเต็ม ≠ ยกเลิกทัวร์ ต้องหาโรงแรมระดับเดียวกันทดแทนเสมอ
5. **Multi-property Tour** — 1 Tour = หลายโรงแรม หลายเมือง จัดการแยกกันได้อิสระต่อคืน
6. **Mirror Booking** — Booking มีหน้าตาเหมือน Quotation ที่ Agent เห็น เพื่อลด confusion
